컴포넌트 작업 함정

T1 패널 내부 요소를 raw 프레임으로 새로 그리는 실수
❌ 일어난 일

Chat History Panel 안에 Button, Text Field, History Item을 컴포넌트 인스턴스가 아닌 raw 프레임으로 직접 그려서 채움. 겉으로는 똑같아 보이지만 컴포넌트 업데이트가 패널에 반영되지 않음.

✅ 올바른 방법

작업 시작 전 design-system/components.md를 확인해 이미 만들어진 컴포넌트 목록을 파악한다. 존재하는 컴포넌트는 반드시 createInstance()로 인스턴스를 생성해 사용한다.

원인

패널 작업 요청이 왔을 때 어떤 컴포넌트가 존재하는지 먼저 확인하지 않고 바로 레이아웃을 구성하기 시작함. 빠르게 구현하려다 싱크가 깨지는 더 큰 문제를 만들어냄.

T2 컴포넌트 세트의 일부 variant만 수정하는 실수
❌ 일어난 일

Chat History Panel을 수정할 때 사용자가 전달한 URL의 node-id(State=default)만 확인하고 작업함. State=empty variant가 존재했지만 그대로 raw 상태로 남겨짐.

✅ 올바른 방법

특정 node-id를 받았더라도 부모 프레임 metadata를 먼저 조회해 variant 전체 목록을 확인한다. 이후 모든 variant에 동일한 작업을 적용한다.

작업 순서
1. get_metadata(parentFrameId) → variant 목록 확인
2. 모든 variant ID 수집
3. 각 variant에 동일 작업 반복 적용

API 기술 함정

T3 vectorPaths — 오픈소스 SVG 경로 직접 주입 불가

Figma Plugin API의 vectorPaths는 MCP 환경에서 SVG 경로 커맨드의 일부를 지원하지 않는다. Simple Icons, LobeHub 등에서 가져온 실제 브랜드 SVG 경로를 코드로 직접 주입하면 파싱 오류가 발생한다.

지원 안 되는 SVG 커맨드
h / H v / V a / A t / T + 소수점 좌표 직접 파싱 실패
✅ 대안
  • LobeHub Icons에서 SVG를 직접 Figma에 드래그해 넣는다
  • 오픈소스 SVG를 Figma에서 직접 가져와 컴포넌트에 적용한다
  • 코드로 만들어야 할 경우 createStar(), createPolygon() 등 내장 shape API를 쓴다
T4 createNodeFromSvgAsync — MCP 환경에서 미지원

Figma Plugin API 공식 문서에는 figma.createNodeFromSvgAsync(svg)가 존재하지만, MCP(Claude Code) 환경에서는 이 메서드가 없다.

// TypeError: figma.createNodeFromSvgAsync: no such property
const node = await figma.createNodeFromSvgAsync(svgString);
✅ 대안

SVG 가져오기는 사용자 직접 드래그 방식을 권장. 코드에서는 createStar(), createPolygon(), createRectangle() 등의 기본 shape API로 대체한다.

T5 combineAsVariants — 자동 배치 안 됨, 위치 직접 설정 필요

figma.combineAsVariants(components, page)를 호출하면 모든 variant가 좌표 (0, 0)에 겹쳐진다. combine 전에 각 컴포넌트 위치를 지정해도 무시된다.

// ❌ combine 전 위치 지정 → 무시됨
comp.x = 100; comp.y = 200;
const set = figma.combineAsVariants([...], page);

// ✅ combine 후 자식 위치 직접 설정
set.layoutMode = 'NONE';
for (const child of set.children) {'{'}
  child.x = colX[size]; child.y = rowY[model];
{'}'}
set.resize(totalW, totalH);
✅ 올바른 순서
  1. 컴포넌트를 생성 (위치는 어디든 상관없음)
  2. combineAsVariants() 호출
  3. 세트에 layoutMode = 'NONE' 설정
  4. 각 자식 노드에 x, y 직접 지정
  5. 세트 전체 resize(w, h)

운영 함정

T6 작업 기록을 AI 도구 외부 메모리에 저장하는 실수
❌ 일어난 일

작업 중 배운 규칙(컴포넌트 인스턴스 필수 사용)을 ~/.claude/projects/.../memory/ 경로에 별도 파일로 저장. 프로젝트 문서와 분리되어 팀이 볼 수 없고, 도구가 바뀌면 사라짐.

✅ 올바른 방법

작업 중 배운 규칙은 반드시 프로젝트 문서에 기록한다. 규칙 → design-system/rules.md, 결정 이유 → design-system/decisions.md, AI 작업 원칙 → docs/agent-rules.md.

핵심 원칙

AI 도구 자체의 메모리는 세션 간 신뢰성이 낮고, 도구가 바뀌면 사라진다. 프로젝트 폴더 안의 문서가 유일한 신뢰 가능한 기록 공간이다. 이 원칙은 어떤 AI 도구를 쓰든 동일하게 적용된다.
