01

이 문서의 목적

이 가이드는 프로덕트 디자이너가 앞으로 제품 디자인 과정에서 AI를 활용할 때, 어떤 기준으로 AI Workflow 문서와 운영 문서를 구축해나가면 좋을지 정리한 가이드입니다.

📌

이 가이드의 전제 환경 — Cursor, Claude Code, Windsurf, GitHub Copilot 같은 AI 코드 에디터에서 작업하는 디자이너를 기준으로 씌어 있습니다. AI 에디터는 프로젝트 내 .md 파일을 자동으로 읽고 컨텍스트로 활용합니다. Figma 단독 워크플로나 AI 채팅만 쓰는 경우와는 맞지 않을 수 있어요.

핵심 목적은 단순히 "AI에게 프롬프트를 잘 쓰는 법"이 아닙니다. AI가 디자인 시스템, 제품 맥락, UX 의도, 컴포넌트 사용 기준을 이해하고 일관된 결과물을 만들 수 있도록 작업 기준과 운영 구조를 문서화하는 것이 목적입니다.

앞으로 만들 문서의 역할

  • AI가 제품과 디자인 시스템을 이해하는 기준 문서
  • 디자이너가 AI에게 작업을 맡길 때 참고하는 운영 문서
  • 프론트 개발자와 협업할 때 기준이 되는 커뮤니케이션 문서
  • 디자인 결과물을 검토하고 QA하는 체크리스트
  • 반복 작업을 프롬프트/명령어/워크플로로 표준화하는 문서

현장 이슈와 AI 역할 구분

AI Workflow를 구축하기 전에, 현재 디자인 업무에서 실제로 비효율이 있는 지점을 먼저 파악해야 합니다. 그 중에서 AI가 해결할 수 있는 것과 사람이 직접 판단해야 하는 것을 구분하는 것이 출발점입니다.

기존 업무에서 부담이 큰 지점

화면화 / 제작
  • 기획서를 화면으로 다시 해석해야 함 — 기획 내용이 바로 개발 가능한 화면 형태가 아님
  • 요구사항 변경으로 반복 수정이 많음 — 특히 초기 제품, 신규 기능
  • 디자이너의 역할 범위가 과도하게 넓어짐 — 화면 기획부터 QA까지 연속 담당
전달 / 검수
  • 개발 핸드오프가 수작업에 가까움 — 컴포넌트, 상태값, 예외 케이스 별도 정리 필요
  • 구현 이후 디자인 QA 부담이 큼 — 간격, 정렬, 반응형, 브랜드 톤 재확인
  • 핵심 판단보다 화면화/전달/검수에 시간이 더 많이 쓰임

AI 도입 시 새로 생기는 이슈

  • AI 결과물의 품질 편차가 큼 — 브랜드 톤, UX 일관성, 디테일 품질이 흔들릴 수 있음
  • AI 활용 기준이 없음 — 어떤 화면을 맡겨도 되는지, 어떤 부분은 직접 판단해야 하는지 기준 부재
  • 히스토리 관리가 어려워짐 — Figma, 코드, 문서가 분산될 경우 최신 기준과 결정 이유 추적 어려움

AI vs 인간 — 잘하는 일과 한계

구분 잘하는 일 부담이 크거나 한계인 일
AI 구조화, 반복 생성, 변형안 제작, 체크리스트화, 누락 점검, 문서화 자동화 제품 맥락 이해, 브랜드 감도 판단, 복잡한 UX 의사결정, 비즈니스/정책 판단, 최종 책임
인간 문제 정의, 제품 방향 결정, 브랜드 감도 판단, UX 맥락 해석, 우선순위 결정, 최종 품질 판단 빈 화면에서 초안 만들기, 반복 UI 대량 제작, 모든 상태값 기억, 핸드오프 항목 정리, QA 반복 검수, 변경 이력 정리

AI로 해결 가능한 업무

화면 초안
기획서 기반 1차 구조화
기획서를 AI에 입력해 화면 목적, 주요 섹션, 상태값, 예외 케이스 초안을 생성. 디자이너는 UX 흐름과 브랜드 기준으로 검토/수정.
반복 UI
패턴화된 UI 빠른 생성
카드, 리스트, 폼, 빈 상태, 에러/로딩 상태처럼 구조가 반복되는 UI. 컴포넌트 규칙을 프롬프트에 포함해 디자인 시스템 안에서 생성.
핸드오프
체크리스트 자동화
화면 기획서나 시안을 AI에 입력해 컴포넌트, 상태값, 반응형, 문구, API 필요 여부 등 개발 전달 항목을 자동 추출.
QA
사전 점검 보조
구현 화면 기준으로 누락된 상태, 불명확한 문구, 반응형 이슈, 디자인 시스템 위반 가능성을 AI가 1차 점검. 최종 QA는 디자이너 직접 진행.
브랜드
결과물 편차 감소
브랜드 톤앤매너, 문구 스타일, 피해야 할 느낌을 가이드로 정리해 프롬프트에 포함. AI 결과물의 브랜드 편차를 줄이고, 맞지 않는 경우 재생성.
문서화
재사용 가능한 프로세스
잘 작동한 프롬프트와 실패한 프롬프트를 기록해 팀 문서로 정리. 반복 가능한 업무 방식으로 체계화.

AI로 해결이 어려운 업무

⚠️

제품 방향성, 핵심 UX 판단, 우선순위 결정은 AI가 대신할 수 없습니다.
B2C 제품에서 중요한 브랜드 첫인상, 감정 설계, 신뢰감 형성은 디자이너가 직접 판단해야 합니다.
복잡한 사용자 맥락, 정책, 비즈니스 조건이 얽힌 화면은 AI 초안만으로 결정하기 어렵고, 조직 전체의 협업 방식, 의사결정 구조는 AI 도입만으로 해결되지 않습니다.

아직 해결 안 된 문제들

히스토리 관리
  • 어떤 수정본이 최신인지 추적 어려움
  • AI가 결과를 계속 뒤집어도 결정 이력이 안 남음
  • Figma처럼 시각적 비교가 안 됨 — Git + 문서 조합 필요
협업 충돌
  • 디자이너 수정 → 개발자 수정 → AI 재생성 → 코드 충돌 반복
  • SOT(Source of Truth — 팀의 최신 기준이 되는 단일 출처) 가 어느 쪽인지 불명확하면 구조 붕괴
  • 피그마와 코드 라이브러리 싱크 유지가 어려움
Ownership 감각 변화
  • "AI가 추천해서 만든 것"이 반복되면 설계 주도권이 흐려짐
  • 브랜드 감각, 인터랙션 디테일은 사람이 강하게 개입해야 유지됨
일정 예측 어려움
  • 실험적 작업은 일정 산정이 어려움 — 리스크 허용 범위를 팀이 사전 합의해야 함
  • 프롬프팅 비용이 예상보다 클 수 있음 — 맥락이 많을수록 사람이 더 빠른 경우도 있음
💡

현재 가장 현실적인 접근은 혼합형입니다. Figma는 탐색·확정·히스토리 도구로 유지하고, AI + 코드는 반복 UI 제작·수정·검증에 집중하는 방식. 완벽한 자동화보다 빠른 반복과 공통 언어 기반 협업 구조를 만드는 것이 핵심입니다.

최종 Goal: 디자이너에게 집중되어 있던 화면 구체화, 개발 핸드오프, QA 부담을 줄이고 — AI를 활용할 수 있는 업무와 디자이너가 반드시 직접 판단해야 하는 업무를 구분하는 기준을 만든다. B2C 제품의 브랜드 품질과 핵심 경험 완성도는 유지하면서, 반복 작업과 수정 작업의 속도를 높인다.

전체 문서 레이어 구조

운영 문서는 크게 세 가지 레이어로 나눕니다. 이 세 레이어는 단순 나열이 아니라 상하 관계를 가집니다. UX가 최상위 판단 기준이 되고, 디자인 시스템은 그 방향을 일관되게 구현하는 수단이며, AI Workflow는 이 두 레이어를 더 효율적으로 운영하는 도구입니다.

1. Product / UX Context — 판단 기준. 왜 이 화면이 이렇게 설계되어야 하는지, 어떤 사용자 행동을 이끌어야 하는지
↓ 이 기준을 시각 언어로 구현
2. Design System Context — 구현 기준. tokens, components, Storybook(QA), 접근성. UX 방향이 확정된 후 시스템화
↓ 이 시스템을 효율적으로 운영
3. AI Workflow / Operation — 운영 도구. prompts, agent rules, review flow, QA, human approval
💡

스타일이나 화면 방향성이 확정되기 전까지는 AI로 바로 수정하기보다, Figma 또는 탐색 단계에서 방향성을 먼저 결정하고 확정된 방향을 디자인 시스템 레이어로 가져오는 흐름이 더 안정적입니다.

제품 단계별 AI 활용 전략

AI 디자인 프로세스는 "Figma를 대체하느냐"의 문제가 아닙니다. 제품 단계와 성격에 따라 어디까지 AI에 맡길지를 정하는 문제입니다.

Figma의 역할은 팀 구조와 프로세스에 따라 다릅니다. 이해관계자 합의가 중요한 팀은 Figma를 최종 명세 문서로 계속 유지하고, 빠른 구현이 중심인 팀은 탐색·합의용으로만 쓰기도 합니다. 중요한 건 Figma냐 AI냐가 아니라, 팀이 합의한 방식으로 일관되게 운영하는 것입니다.

📌

아래 흐름은 하나의 예시입니다. 팀의 디자이너/개발자 구성, 이해관계자 환경, 제품 성격에 따라 Figma의 비중은 달라질 수 있습니다.

초기 방향/플로우 — Figma (탐색, 이해관계자 합의) 또는 AI 텍스트 스케치
반복 화면/구현 — AI + 코드 (생산, 수정)
디테일 수정/QA — 실제 프론트 (검증, 품질)
히스토리 — Git (결정 기록, 이력 관리)

AI 중심 프로세스가 더 나은 경우

AI 중심이 유리한 조건
  • 내부 툴, SaaS, 어드민, 반복 UI가 많은 서비스
  • 빠르게 만들고 실제 화면에서 바로 고치는 조직
  • 디자이너가 CSS/프론트 구조를 어느 정도 이해하는 팀
  • Git/문서/QA 룰을 잡을 수 있는 팀
  • 완벽한 시안보다 빠른 검증이 중요한 단계
AI 중심이 오히려 혼란을 키우는 조건
  • 브랜드 감도나 비주얼 완성도가 중요한 B2C
  • 이해관계자가 많아서 시각적 합의가 중요한 조직
  • 디자이너와 개발자 역할이 엄격히 나뉜 팀
  • Git, 코드 리뷰, QA 체계가 약한 팀
  • 디자이너가 실제 구현물을 만질 준비가 안 된 팀

1 → 10

  • 디자인 시스템 + 컴포넌트 룰
  • 디자인 토큰 + 상태값 정의
  • 히스토리 관리
  • AI 생성 가드레일 (agent-rules.md)

안정성 우선 — AI는 creator에서 accelerator로

제품이 커지면 속도보다 안정성이 중요해집니다. 디자인 문제가 아니라 운영·조직·거버넌스 문제가 됩니다.

이 단계의 주요 이슈
접근성, 다크모드, 다국어, 예외 케이스, 권한/상태별 화면, 의사결정 기록, 코드 품질

이전
AI = creator (메인 제작자)
이후

QA 보조, 리팩토링, 마이그레이션, variation 생성, 문서화 자동화에 더 적합합니다.

💡

디자이너는 단순히 화면을 그리는 사람이 아니라, AI가 만든 결과물이 제품답게 보이고 작동하도록 품질을 관리하는 사람에 가까워집니다.

AI 시대의 디자인 프로세스는 "더 빨리 만드는 방식"이 아니라, 제품 단계에 맞게 속도 · 브랜드 · 일관성 · 안정성의 균형을 다시 설계하는 방식입니다.

AI 에디터 연결 설정

docs/ 폴더를 만들었다면, 이제 AI가 실제로 이 문서들을 읽게 해야 합니다. 문서가 아무리 잘 만들어져 있어도 AI 에디터에 연결되지 않으면 그냥 폴더 안의 파일입니다.

Figma MCP 연결 설정

연결 전 확인할 것

계정 / 권한
  • OAuth를 승인하는 Figma 계정이 실제 작업 파일에 접근 가능한지 확인
  • 팀 파일을 다룰 경우 Dev 또는 Full seat 권한이 필요한지 확인
  • 공용 계정으로 연결한다면 누가, 어떤 범위까지 접근 가능한지 기록
작업 대상
  • 파일 URL뿐 아니라 작업할 page 또는 node URL까지 준비
  • 여러 페이지가 있는 파일은 AI가 첫 페이지를 기준으로 작업하지 않도록 명시
  • Variables, styles, components를 읽을 수 있는지 첫 테스트에서 확인

기본 연결 흐름

1. AI 도구에서 Figma MCP 연결 추가
Claude Code, Cursor, Codex 등 사용하는 MCP 클라이언트의 공식 설정 화면에서 Figma MCP를 추가합니다.
2. Figma OAuth 승인 계정 확인
승인한 계정이 볼 수 있는 파일만 AI가 접근할 수 있습니다. 팀 파일이면 개인 계정/공용 계정 선택을 먼저 정합니다.
3. Desktop MCP를 쓰는 경우 Dev Mode에서 서버 활성화
Figma 데스크탑 앱에서 대상 파일을 열고 Dev Mode로 전환한 뒤 MCP 서버가 켜져 있는지 확인합니다.
4. IDE/터미널에서 MCP 상태 확인
Claude Code라면 /mcp 같은 서버 상태 화면에서 연결 실패 여부를 확인하고 필요 시 reconnect합니다. Cursor/Codex도 MCP 서버 목록에서 연결 상태를 먼저 봅니다.
5. 첫 테스트는 읽기 작업으로 시작
Figma 링크를 주고 "이 파일의 page 목록, 선택한 frame의 metadata, variables를 읽을 수 있는지 확인해줘"처럼 요청합니다.

첫 연결 테스트 체크리스트

읽기 테스트
  • Figma 파일 URL을 주고 page 목록을 읽을 수 있는지 확인
  • 작업할 frame/node URL을 주고 metadata를 읽을 수 있는지 확인
  • Variables, text styles, component 목록을 읽을 수 있는지 확인
쓰기 테스트
  • 운영 파일이 아닌 테스트 파일에서 작은 text 또는 rectangle을 생성
  • 여러 page가 있는 파일이면 작업 page를 먼저 지정할 수 있는지 확인
  • 실제 디자인 시스템 파일에 쓰기 전에는 반드시 읽기 결과를 먼저 검토
첫 테스트 프롬프트
이 Figma 파일에 접근할 수 있는지 먼저 확인해줘.
1. page 목록을 읽어줘.
2. 이 node URL의 metadata를 요약해줘.
3. 이 파일에서 variables와 text styles를 읽을 수 있는지 확인해줘.
아직 캔버스에는 아무것도 만들지 말고, 읽기 가능 여부만 알려줘.
그대로 적으면 위험한 말
  • "신형 MCP는 항상 Figma 서버에서만 실행된다"
  • "Variable 컬러는 AI가 못 읽는다"
  • "로컬 폰트는 무조건 사용할 수 없다"
정확한 표현
  • Remote MCP와 Desktop MCP는 설정 방식이 다르다
  • Variables는 연결 권한과 파일 구조가 맞으면 읽을 수 있다
  • 폰트 접근은 Remote/Desktop 방식, Figma 업로드 여부에 따라 달라질 수 있다
⚠️

MCP 연결 정보도 운영 문서에 남깁니다. 어떤 계정으로 승인했는지, 어떤 파일을 테스트했는지, 연결 실패 시 어디서 reconnect하는지 기록해두면 공용 계정·공용 파일에서 같은 시행착오를 반복하지 않습니다.

컴포넌트 작업 함정

T1 패널 내부 요소를 raw 프레임으로 새로 그리는 실수
❌ 일어난 일

Chat History Panel 안에 Button, Text Field, History Item을 컴포넌트 인스턴스가 아닌 raw 프레임으로 직접 그려서 채움. 겉으로는 똑같아 보이지만 컴포넌트 업데이트가 패널에 반영되지 않음.

✅ 올바른 방법

작업 시작 전 design-system/components.md를 확인해 이미 만들어진 컴포넌트 목록을 파악한다. 존재하는 컴포넌트는 반드시 createInstance()로 인스턴스를 생성해 사용한다.

원인

패널 작업 요청이 왔을 때 어떤 컴포넌트가 존재하는지 먼저 확인하지 않고 바로 레이아웃을 구성하기 시작함. 빠르게 구현하려다 싱크가 깨지는 더 큰 문제를 만들어냄.

T2 컴포넌트 세트의 일부 variant만 수정하는 실수
❌ 일어난 일

Chat History Panel을 수정할 때 사용자가 전달한 URL의 node-id(State=default)만 확인하고 작업함. State=empty variant가 존재했지만 그대로 raw 상태로 남겨짐.

✅ 올바른 방법

특정 node-id를 받았더라도 부모 프레임 metadata를 먼저 조회해 variant 전체 목록을 확인한다. 이후 모든 variant에 동일한 작업을 적용한다.

작업 순서
1. get_metadata(parentFrameId) → variant 목록 확인
2. 모든 variant ID 수집
3. 각 variant에 동일 작업 반복 적용

API 기술 함정

T3 vectorPaths — 오픈소스 SVG 경로 직접 주입 불가

Figma Plugin API의 vectorPaths는 MCP 환경에서 SVG 경로 커맨드의 일부를 지원하지 않는다. Simple Icons, LobeHub 등에서 가져온 실제 브랜드 SVG 경로를 코드로 직접 주입하면 파싱 오류가 발생한다.

지원 안 되는 SVG 커맨드
h / H v / V a / A t / T + 소수점 좌표 직접 파싱 실패
✅ 대안
  • LobeHub Icons에서 SVG를 직접 Figma에 드래그해 넣는다
  • 오픈소스 SVG를 Figma에서 직접 가져와 컴포넌트에 적용한다
  • 코드로 만들어야 할 경우 createStar(), createPolygon() 등 내장 shape API를 쓴다
T4 createNodeFromSvgAsync — MCP 환경에서 미지원

Figma Plugin API 공식 문서에는 figma.createNodeFromSvgAsync(svg)가 존재하지만, MCP(Claude Code) 환경에서는 이 메서드가 없다.

// TypeError: figma.createNodeFromSvgAsync: no such property
const node = await figma.createNodeFromSvgAsync(svgString);
✅ 대안

SVG 가져오기는 사용자 직접 드래그 방식을 권장. 코드에서는 createStar(), createPolygon(), createRectangle() 등의 기본 shape API로 대체한다.

T5 combineAsVariants — 자동 배치 안 됨, 위치 직접 설정 필요

figma.combineAsVariants(components, page)를 호출하면 모든 variant가 좌표 (0, 0)에 겹쳐진다. combine 전에 각 컴포넌트 위치를 지정해도 무시된다.

// ❌ combine 전 위치 지정 → 무시됨
comp.x = 100; comp.y = 200;
const set = figma.combineAsVariants([...], page);

// ✅ combine 후 자식 위치 직접 설정
set.layoutMode = 'NONE';
for (const child of set.children) {'{'}
  child.x = colX[size]; child.y = rowY[model];
{'}'}
set.resize(totalW, totalH);
✅ 올바른 순서
  1. 컴포넌트를 생성 (위치는 어디든 상관없음)
  2. combineAsVariants() 호출
  3. 세트에 layoutMode = 'NONE' 설정
  4. 각 자식 노드에 x, y 직접 지정
  5. 세트 전체 resize(w, h)

운영 함정

T6 작업 기록을 AI 도구 외부 메모리에 저장하는 실수
❌ 일어난 일

작업 중 배운 규칙(컴포넌트 인스턴스 필수 사용)을 ~/.claude/projects/.../memory/ 경로에 별도 파일로 저장. 프로젝트 문서와 분리되어 팀이 볼 수 없고, 도구가 바뀌면 사라짐.

✅ 올바른 방법

작업 중 배운 규칙은 반드시 프로젝트 문서에 기록한다. 규칙 → design-system/rules.md, 결정 이유 → design-system/decisions.md, AI 작업 원칙 → docs/agent-rules.md.

핵심 원칙

AI 도구 자체의 메모리는 세션 간 신뢰성이 낮고, 도구가 바뀌면 사라진다. 프로젝트 폴더 안의 문서가 유일한 신뢰 가능한 기록 공간이다. 이 원칙은 어떤 AI 도구를 쓰든 동일하게 적용된다.

환경 / 연결 함정

T7 작업 페이지를 지정하지 않아 첫 번째 페이지만 바라보는 실수

Figma 파일에 여러 페이지가 있을 때, 에이전트가 기본 페이지나 첫 번째 페이지 기준으로 노드를 찾는 경우가 있다. 사용자가 특정 페이지 링크를 주었더라도 코드 실행 시 페이지 컨텍스트를 명시하지 않으면 엉뚱한 페이지에서 작업할 수 있다.

✅ 대안

작업 시작 시 전달받은 node-id가 속한 page를 확인하고, 필요한 경우 await figma.setCurrentPageAsync(page)를 먼저 실행한다. 페이지 이동이 필요한 스크립트에는 이 처리를 공통 유틸처럼 넣는다.

T8 로컬 폰트 접근 여부를 단정하는 실수

Figma AI나 원격 플러그인 방식은 내 컴퓨터의 로컬 폰트를 직접 쓰지 못할 수 있다. 반대로 Figma 데스크탑 앱과 연결된 로컬 서버 방식에서는 데스크탑 앱이 인식한 폰트를 사용할 수 있는 경우가 있어, “항상 불가”처럼 단정하면 안 된다.

✅ 대안

팀에서 쓰는 폰트는 Figma 팀 폰트 또는 클라우드에서 접근 가능한 폰트로 등록한다. AI에게는 폰트명, 용도, 대체 폰트, 지원 언어를 문서로 제공하고, 실행 전 figma.loadFontAsync() 가능 여부를 확인한다.

T9 Variables를 못 읽는다고 결론내리는 실수

연결 방식이나 권한, 변수 구조에 따라 AI가 Variables를 충분히 읽지 못하는 것처럼 보일 수 있다. 하지만 MCP가 올바르게 연결되어 있고 필요한 권한이 있으면 컬렉션, 모드, 오타, semantic token 부족 같은 구조 문제까지 감사할 수 있다.

✅ 대안

“Variables 못 읽음”으로 기록하지 말고, 연결 방식과 권한을 먼저 확인한다. Semantic 컬렉션에는 bg, text, border, icon, status처럼 목적 기반 토큰을 충분히 만들고, mode 이름은 light/dark처럼 명확하게 둔다.

도구별 접근 방식

같은 탐색 프로세스를 두 가지 방식으로 실행할 수 있습니다. 어떤 도구를 쓰든 최종적으로 도달하는 결과물 — 문서로 전환할 수 있는 기준 — 은 동일합니다.

Figma 기반 접근
  • Frame 단위로 화면 구조를 시각적으로 탐색
  • Color Style / Text Style로 token 후보 자연스럽게 발견
  • Auto layout으로 spacing 패턴 파악
  • Component 지정하면서 반복 패턴 추출
  • 시안을 쌓으면서 variant 필요성 발견
  • Prototype으로 UX 흐름 검증
AI + 텍스트 기반 접근
  • 레퍼런스 이미지 수집 후 AI에게 스타일 분석 요청
  • 마크다운/노션으로 화면 구조를 텍스트로 스케치
  • AI에게 초안 코드/화면 생성 요청
  • 생성된 결과물을 보면서 패턴 발견
  • 반복되는 구조를 바로 component-usage.md로 정리
  • AI와 대화하면서 UX 의도 언어화

Product / UX Context 문서 만들기

AI가 시안을 만들려면 먼저 제품의 방향과 UX 판단 기준을 알아야 합니다. 이 문서는 디자인 시스템보다 상위에 있는 문서입니다.

product-context.md 예시

product-context.md
# Product Context

## 제품 목적
사용자가 복잡한 정보를 빠르게 이해하고,
다음 행동을 쉽게 결정할 수 있도록 돕는다.

## 주요 사용자
- 신규 사용자
- 반복 사용 사용자
- 관리자/운영자

## 제품 톤
- 차분함
- 명확함
- 신뢰감
- 과도하게 귀엽거나 장식적이지 않음

## AI 작업 시 전제
- 화면은 정보 위계가 명확해야 한다.
- 한 화면에 너무 많은 선택지를 노출하지 않는다.
- primary action은 명확하게 드러나야 한다.

ux-principles.md 예시

ux-principles.md
# UX Principles

## Hierarchy
사용자가 가장 먼저 봐야 하는 정보와 행동을 명확히 구분한다.
primary action은 한 화면에서 하나를 원칙으로 한다.

## Density
초기 사용자를 위한 화면은 낮은 정보 밀도를 유지하고,
반복 사용자를 위한 관리 화면은 높은 정보 밀도를 허용한다.

## Cognitive Load
한 화면에서 사용자가 판단해야 하는 선택지를 과도하게 늘리지 않는다.
입력 단계는 가능한 한 쪼개고, 한 번에 요구하는 정보량을 최소화한다.

## Tone
제품이 사용자에게 주는 말투/인상/감정을 일관되게 유지한다.
장식적이거나 과도하게 가벼운 느낌은 브랜드 기준에서 판단한다.

## Motion
모션은 사용자의 주의를 끌기 위한 수단이 아니라,
전환/피드백/상태 변화를 명확히 전달할 때만 사용한다.

## Composition
화면 구성은 여백, 정렬, 시선 흐름이 자연스러워야 한다.
불필요한 UI 요소를 줄여 핵심 정보가 도드라지게 한다.

## UX Intent
모든 주요 화면과 컴포넌트는 사용자를 어떤 행동으로 이끌기 위한 것인지
설명할 수 있어야 한다. 설명할 수 없는 UI 요소는 추가하지 않는다.
✅

UX context가 있으면 AI가 "이 화면이 왜 이렇게 설계되어야 하는지"를 반영할 수 있습니다. 단순히 보기 좋은 UI가 아닌 제품 의도에 맞는 결과물을 만들게 됩니다.

Design System Context 문서 만들기

AI가 일관된 화면을 만들려면 디자인 시스템을 알아야 합니다. 단순히 token과 component 목록을 적는 것이 아니라, 언제/왜/어떻게 써야 하는지까지 포함해야 합니다.

component-usage.md — Button 예시

component-usage.md
# Button

## 목적
사용자가 명확한 행동을 실행하도록 돕는 컴포넌트다.

## UX intent
사용자가 다음 행동을 명확히 인지하고 실행하게 한다.

## Hierarchy
- primary action은 한 화면에서 과도하게 중복되지 않도록 한다.
- 핵심 행동(저장/완료/다음 단계)에는 primary를 사용한다.
- 보조 행동에는 secondary 또는 text link를 사용한다.
- destructive action은 삭제, 되돌릴 수 없는 행동에만 사용한다.

## Density
- 모바일에서는 충분한 터치 영역을 유지한다. (최소 44×44px)
- 좁은 화면에서 버튼이 겹치거나 잘리지 않도록 한다.

## 사용하지 않는 경우
- 단순 이동 링크에는 Button 대신 Link를 우선 고려한다.
- 정보 표시 목적으로 Button을 사용하지 않는다.

## Accessibility
- 텍스트 contrast는 WCAG AA 기준 4.5:1 이상 유지한다.
- large text / UI graphical object는 3:1 이상.
- focus 상태는 시각적으로 명확해야 한다. (focus ring 식별 가능)
- keyboard interaction을 지원해야 한다. (Enter, Space)
- disabled 상태에서도 contrast 기준을 참고한다.

## Storybook 확인 상태
default / hover / focus / disabled / loading / long text / mobile / dark mode

이런 문서가 있으면 AI는 버튼을 단순히 "예쁜 사각형"으로 만들지 않고, 제품 내 역할에 맞게 사용할 수 있습니다.

Storybook의 역할

맥락을 이어주는 기록 채널 설계

AI Workflow는 매번 새 프롬프트에서 새로 시작하는 방식이 아니라, 사람과 AI가 같은 업무 맥락을 다시 찾을 수 있는 기록 채널을 함께 운영할 때 안정적으로 이어집니다.

기록 채널의 형태는 하나로 고정하지 않습니다. Notion, Git 기반 Markdown, HTML viewer, issue tracker, wiki 등 팀의 도구 환경에 맞게 선택할 수 있습니다. 중요한 기준은 사람이 읽기 쉽고, AI가 맥락을 회수할 수 있으며, 변경 이력과 실행 규칙으로 이어질 수 있는 구조입니다.

기록 형태 주요 역할 운영 기준
Worklog / Daily log그날의 작업, 판단, 미완료 항목, 함정을 시간순으로 남김다음 세션에서 AI와 사람이 바로 이어받을 수 있어야 함
Notion / Wiki팀이 읽기 쉬운 지식 베이스와 업무 맥락 저장소개인 기록을 그대로 쌓기보다 공유 가능한 단위로 정리
Git 기반 Markdown변경 이력과 코드 작업 맥락을 함께 추적커밋, PR, 파일 변경과 연결 가능해야 함
HTML viewerGit과 Notion 사이에서 사람이 빠르게 훑는 읽기 전용 뷰어데이터를 하드코딩하기보다 기존 기록 구조에서 재사용 가능해야 함
Rules / Playbook반복되는 실수와 운영 원칙을 실행 기준으로 승격작업 기록과 분리해 AI가 매번 따라야 하는 규칙으로 관리

기록에서 실행 기준으로 승격하기

  1. 작업 중 생긴 결정, 실수, 검증 결과, 미완료 항목을 원본 기록 채널에 남깁니다.
  2. 반복될 가능성이 있는 내용은 단순 회고에 묻어두지 않고 rule, playbook, checklist 후보로 표시합니다.
  3. AI가 다음 세션에서도 반드시 따라야 하는 내용은 agent-rules.md, workflow 문서, QA checklist 같은 실행 기준 문서로 승격합니다.
  4. 빠른 처리를 위해 값을 임시로 박아 넣기보다, 기존 구조와 재사용 가능한 규칙을 먼저 확인합니다.

Agent Rules 문서 만들기

Agent Rules는 AI가 반드시 지켜야 하는 규칙입니다. AI는 요청을 잘 수행하지만, 문맥이 없으면 쉽게 기준을 벗어납니다.

agent-rules.md 예시

agent-rules.md
# Agent Rules

## 공통 규칙
- 기존 디자인 시스템 기준을 우선 따른다.
- hardcoded color, spacing, font-size 사용을 금지한다.
- 새로운 컴포넌트 전, 기존으로 해결 가능한지 먼저 확인한다.
- public component API는 임의로 변경하지 않는다.
- 접근성 관련 속성은 임의로 제거하지 않는다.
- 변경 이유와 영향 범위를 요약한다.

## 디자인 관련 규칙
- 화면의 primary action은 명확해야 한다.
- 한 화면에서 너무 많은 강조 요소를 만들지 않는다.
- 제품 톤과 맞지 않는 장식적 요소를 추가하지 않는다.
- UX intent가 불명확한 UI 요소는 추가하지 않는다.

## 확인 규칙
- 변경 후 관련 컴포넌트 상태를 확인한다.
- 접근성 기준을 확인한다.
- 반응형에서 깨지는 부분이 없는지 확인한다.

Agent Rules가 방지하는 문제

Token/Component 이탈
  • token 대신 임의 색상 사용
  • 기존 컴포넌트 대신 새 UI 생성
UX 기준 이탈
  • primary button을 여러 개 남발
  • 화면 톤이 제품과 달라짐
접근성 누락
  • 접근성 속성 삭제
  • hover/focus/disabled 상태 누락
범위 초과
  • 요청 범위 이상의 변경
  • 임의 variant 추가

Prompts 문서 만들기

프롬프트 문서는 반복 작업을 빠르게 요청하기 위한 템플릿입니다. 단, product context, design system context, agent rules가 먼저 있어야 좋은 프롬프트가 나옵니다.

처음 프롬프트는 어디서 시작하나

처음부터 완벽한 프롬프트를 쓰려고 하면 오히려 막힙니다. 아래 순서로 점진적으로 채워가세요.

1단계 — 목적만 먼저  → "신규 회원가입 화면 초안 만들어줘"
↓ 결과 보고 부족한 기준 추가
2단계 — 기준 파일 연결  → product-context.md, component-usage.md 참고해서
↓ AI가 반복해서 틀리는 부분 발견
3단계 — 구체 제약 추가  → primary action 1개만, hardcoded style 금지
↓ 결과가 안정되면
4단계 — prompts.md에 저장  → 다음에 같은 작업 시 재사용

완성된 프롬프트의 구조

  1. 작업 목적

  4. 지켜야 할 UX 기준
  5. 금지할 변경
  6. 결과물 형식
  7. 검토 기준

- Button, Card, Text 컴포넌트를 우선 사용합니다.
- hardcoded color/spacing은 사용하지 않습니다.
- 접근성 기준을 충족해야 합니다.

결과물:
- 화면 구조
- 사용 컴포넌트
- UX 의도
- 검토해야 할 상태

디자이너 + AI QA 운영 구조

디자이너가 AI로 간단한 수정과 QA 리포팅을 할 수 있어도, 운영 중인 제품 코드는 디자이너 수정안 작성 → 개발자 검수 → 반영 흐름으로 둡니다. AI가 만든 변경을 바로 배포하지 않고, 원본 코드와 분리된 변경 단위로 확인해야 운영 리스크가 작아집니다.

1. 디자이너가 문제 지점 지정 — Figma 링크, 개발 화면 URL, 디바이스/브라우저, 기대 결과와 현재 결과를 함께 기록
↓
2. AI가 수정안 또는 티켓 초안 생성 — 코드 수정, 원인 추정, 재현 조건, QA 체크 항목을 초안으로 만든다
↓
3. 디자이너가 디자인 QA 판단 — 시각 품질, UX 의도, 컴포넌트/토큰 기준 위반 여부를 확인한다
↓
4. 개발자가 코드 검수 후 반영 — 기능 영향, 코드 구조, 라이브러리 재사용성을 개발자가 최종 확인한다

QA R&R을 먼저 나눠야 하는 영역

구분 디자이너가 주로 확인할 것 개발자와 확인할 것
디자인 QA 간격, 정렬, 색상, 타이포그래피, 반응형 시각 품질, Figma 의도와의 차이 토큰/컴포넌트가 코드에서 어떤 방식으로 연결되는지
기능 QA 사용자 흐름상 어색한 상태, 문구, 피드백 타이밍 권한, 데이터 분기, API 상태, 예외 케이스, 회귀 영향
자동화 가능 영역 스크린샷 표시, 기대/현재 결과 정리, 티켓 초안, 반복 체크리스트 완전 자동 티켓 생성보다 사람이 선택한 차이만 티켓화하는 반자동 흐름

도구 / 작업 선택 가이드

AI 도구는 기능 목록으로 외우기보다, 실제 업무 흐름 안에서 선택하는 것이 좋습니다. 먼저 지금 하려는 일이 탐색, 정리, 검증, 전달 중 어디에 가까운지 판단하고 그에 맞는 작업 방식을 고릅니다.

핵심 기준: Figma와 AI 도구는 디자인 판단을 대신하는 도구가 아니라, 이미 정한 제품 맥락·UX 원칙·디자인 시스템 기준을 빠르게 적용하고 검토하기 위한 운영 도구입니다.

업무 흐름별 선택 순서

1. 탐색 / 초안 — 화면 구조나 플로우를 빠르게 보고 싶다면 Figma 시안, FigJam 플로우, 코드 스파이크 중 가장 빠른 형식으로 시작
↓
2. 시스템 적용 — 확정에 가까운 화면은 기존 컴포넌트, token, typography, spacing 기준에 맞춰 정리
↓
3. 드리프트 점검 — 로컬 스타일, override, 누락 token, 컴포넌트 이탈을 체크리스트나 AI 리뷰로 확인
↓
4. 스펙화 — 컴포넌트 릴리즈나 핸드오프 전에는 구조, 상태, 색상, 접근성 기준을 문서화
↓
5. 전달 / 동기화 — 코드 컴포넌트 연결, 토큰 차이, QA 결과를 개발팀이 확인 가능한 형식으로 정리

디자이너가 자주 만나는 상황별 추천

상황 먼저 할 일 다음에 이어질 작업
신규 기능 화면을 빠르게 구체화해야 한다 제품 맥락과 UX 원칙 기준으로 초안 생성 product-context.md, ux-principles.md 기준으로 초안을 만들고, Figma에서 방향성을 좁힙니다.
기존 화면이 디자인 시스템과 어긋나 보인다 컴포넌트, token, local style 이탈 점검 문제 지점을 확인한 뒤 component/token 기준에 맞게 정리합니다.
컴포넌트 업데이트를 개발팀에 넘겨야 한다 상태, variant, 접근성, 사용 조건 문서화 필요에 따라 구조, 색상, 문구, 상태값, 접근성 기준을 세부 스펙으로 보강합니다.
정책, 상태, 예외 케이스가 복잡하다 플로우 다이어그램으로 먼저 정리 FigJam에서 플로우를 합의한 뒤 화면 설계와 QA 체크리스트로 연결합니다.
코드와 Figma 기준이 서로 다를 수 있다 token/컴포넌트 drift 확인 토큰 drift를 확인하고, 변경 영향도가 크면 review / release 문서에 남깁니다.
디자인 리뷰나 공유 자료가 필요하다 결정 배경과 비교안을 공유 자료로 정리 결정 배경, 비교안, QA 결과를 슬라이드로 정리해 팀 공유 자료로 만듭니다.

## Figma / AI 작업 선택 기준
- 탐색 단계에서는 Figma 시안, FigJam 플로우, 코드 스파이크 중 가장 빠른 방식으로 방향을 본다.

방향 이탈(Drift) 감지 신호

AI를 쓰면 더 빨리 만들 수 있지만, 더 빨리 엉뚱한 방향으로 갈 수도 있습니다. 리뷰 막판에 품질을 고치는 것과 별개로, 작업 중간에 방향이 틀어지고 있는 신호를 먼저 잡아야 합니다.

신호 의미 대응 액션
피드백이 같은 질문을 맴돈다 디테일 문제가 아니라 방향 옵션이 부족한 상태 세부 조정을 멈추고 Figma 시안 또는 코드 스파이크 중 가장 빠른 방식으로 2~3개 대안을 새로 만든다
뭔가 이상하지만 설명하지 못한다 시각 품질, 정보 위계, 제품 톤이 섞여 감각적으로 어긋난 상태 Figma AI 에이전트나 감사 체크리스트로 캔버스 전체를 점검한다
빌드가 초기 제안에서 멀어진다 구현 과정에서 원래 UX 의도나 비교 기준이 흐려진 상태 원본 시안과 구현 화면을 나란히 두고 차이를 명시한다
팀이 방향을 고르지 못한다 선택 기준보다 취향 논의가 앞서는 상태 각 안이 얻는 것과 포기하는 것을 적고, 실제 작동 가능한 대안으로 비교한다

방향 확정 전 체크

1. 옵션을 나란히 비교 — 하나씩 말로 설명하지 말고 같은 조건에서 비교한다
↓
2. 트레이드오프를 명시 — 무엇이 좋아졌고 무엇을 포기하는지 적는다
↓
3. Figma 프로토타입 또는 코드로 현실화 — 실제 인터랙션과 제약 조건에서 무너지는지 확인한다
↓
4. 외부인도 이해 가능한지 확인 — 설명 없이 화면만 봐도 의도가 읽히는지 본다

AI로 모션 시안 만들 때의 프롬프트 흐름

모션은 말로만 설명하면 오해가 큽니다. 레퍼런스 영상, 적용할 화면 캡처, 시간 조건을 함께 주고 한 단계씩 좁혀가는 방식이 더 안정적입니다. 목표는 AI가 최종 모션을 대신 확정하는 것이 아니라, 디자이너가 빠르게 비교하고 개발자에게 설명 가능한 산출물로 정리하는 것입니다.

쓸 만한 상황: Figma 프로토타입만으로 원하는 움직임을 만들기 어렵거나, 모션 디자이너 없이 FE와 “이런 느낌”을 공유해야 할 때. 특히 에러 강조, CTA 피드백, 차트 등장, 입력 누락 안내처럼 짧은 UI 모션에 적합합니다.

1. 레퍼런스 확보 — 비슷한 모션을 화면 녹화하거나, 모션 이름을 모르면 AI에게 먼저 이름을 묻는다
↓
2. 적용 화면 캡처 — 실제 적용될 UI를 함께 제공해 위치와 대상 요소를 명확히 한다
↓
3. 순서를 넘버링 — A에서 B로 이동, 100ms 후 강조, 1회만 재생처럼 단계와 시간을 수치로 적는다
↓
4. 조정 가능한 미리보기 요청 — 재생 버튼과 duration/delay 슬라이더를 넣어 디자이너가 감각을 조정하게 한다
↓
5. CSS 코드와 녹화본 전달 — 최종 모션은 CSS 기준 코드, 스크린 녹화, 의도 설명을 함께 개발팀에 넘긴다

항목 넣어야 하는 이유 예시

디자이너가 준비할 것
  • 최종 모션 녹화본 또는 GIF

  • 실제 코드 스택에 맞는 구현 방식 (CSS, React, emotion 등)

2주 실험 실행 계획

AI Workflow 도입의 효과를 확인하기 위한 2주간 실험 계획입니다. 결과물이 아니라 과정이 보이는 자료를 남기는 것이 핵심입니다.

👤

AI 활용 프로젝트 메인 담당자: 프로덕트 디자인팀 담당자가 진행 방식 논의, 회의 주체, 공유, 업무 팔로업을 담당합니다. 파트원이 여럿인 경우 별도 지정합니다.

실험 12단계 프로세스

  1. 현재 디자인 업무에서 부담이 큰 지점 정리
    기획서 확인 → 화면 설계 → 시안 제작 → 핸드오프 → QA 과정을 나열하고 손이 많이 가는 구간 정리. 담당: 디자이너, PM/PO, FE
  2. AI로 해결 가능한 업무와 불가능한 업무 구분
    화면 초안 생성, 반복 UI 제작 vs 브랜드 판단, UX 의사결정, 정책 해석. 담당: 디자이너, PM/PO
  3. 실험할 화면 또는 기능 선정
    신규 기능 초기 화면, 반복 카드 UI, 빈 상태/에러 상태 등 AI 적용 효과를 확인하기 좋은 화면 1~2개 선정. B2C 첫인상 구간은 AI 결과물을 그대로 쓰지 않고 디자이너가 강하게 보정. 담당: 디자이너, PM/PO
  4. 브랜드/UX/디자인 시스템 기준 먼저 정의
    톤앤매너, 컴포넌트 사용 기준, 컬러, 타이포, spacing, 피해야 할 느낌을 간단히 정리. 담당: 디자이너
  5. 기획서를 바탕으로 AI에 1차 화면 구조 생성 요청
    화면 구조, 정보 우선순위, 주요 섹션, 필요한 상태값, 예외 케이스 초안 생성. 완성된 디자인보다 화면 기획 보조와 구조화에 초점. 담당: 디자이너
  6. AI로 화면 초안 또는 UI 변형안 생성
    Claude, Codex 등 사용 가능한 도구로 화면 초안 생성. 반복 UI, 상태 화면, 리스트/카드/폼 구조에 AI 적극 활용. 담당: 디자이너, FE
  7. 디자이너가 UX/브랜드/디자인 시스템 관점으로 1차 보정
    정보 구조, 시선 흐름, 문구, 간격, 브랜드 톤 검수 및 수정. 디자인 시스템 기준을 벗어난 결과물은 수정 또는 재생성. 담당: 디자이너
  8. 프론트 개발자와 구현 가능성 검토
    컴포넌트 구조, 상태 관리, 반응형, API 연동 확인. AI UI가 코드 구조를 복잡하게 만들지 않는지 검토. 담당: FE, 디자이너
  9. 핸드오프 항목 체크리스트화
    컴포넌트 종류, 상태값, 반응형, 에러/로딩/빈 상태, 문구, 접근성, API 필요 여부를 AI가 초안 작성 후 디자이너·FE가 보완. 담당: 디자이너, FE
  10. 실제 구현 화면 기준으로 디자인 QA 진행
    Figma 시안이 아닌 실제 프론트 화면에서 간격, 정렬, 반응형, 문구, 상태, 브랜드 톤 확인. AI로 QA 항목 누락 여부 1차 점검. 담당: 디자이너, PM/PO, FE
  11. 작업 시간과 품질 변화 기록
    화면 초안 제작 시간, 핸드오프 작성 시간, QA 수정 횟수, 브랜드 보정 시간, FE 커뮤니케이션 횟수 기록. 담당: 메인 담당자, 디자이너
  12. 2주 후 결과 공유 및 적용 기준 정리
    AI를 계속 활용할 업무, 제한적으로 활용할 업무, 사용하지 않는 게 나은 업무를 구분. 담당: 전체

AI 도입 시 디자이너 일정 산정 가이드

AI를 활용하면 일부 업무는 빨라지지만, 처음 도입하는 경우 익숙해지는 시간과 반복 검토 비용이 추가됩니다. AI 활용을 전제로 일정을 산정할 때는 다음 기준을 참고하세요.

⚠️

AI 도입 초기에는 일정을 단축하기보다 기존과 동일하게 잡는 것이 현실적입니다. 프롬프팅 비용, 결과물 검토, 보정 작업이 예상보다 클 수 있습니다.

업무 유형별 일정 조정 기준

단축 가능한 업무
  • 반복 UI 제작 — 카드, 리스트, 폼 등 구조가 반복되는 화면: 기존 대비 20~40% 단축 가능
  • 핸드오프 항목 정리 — 컴포넌트·상태값 목록 초안 생성: 30~50% 단축
  • 빈 상태 / 에러 상태 화면 — 패턴이 명확한 상태 화면: 30% 단축
  • 문서 초안 작성 — 체크리스트, 정책 요약, QA 항목: 40~60% 단축
기존 일정 유지 필요 업무
  • 브랜드 첫인상 구간 — 온보딩, 랜딩, 신규 기능 진입 화면: AI 결과물 그대로 사용 불가, 디자이너 직접 검토 필요
  • 복잡한 UX 흐름 — 결제, 인증, 다단계 플로우: AI 의존 시 논리 오류 위험
  • 신규 컴포넌트 설계 — 기존 패턴이 없는 첫 설계: AI 참고는 가능하나 판단은 디자이너 직접

일정 산정 시 추가로 고려할 항목

프롬프트 작성 시간 — 처음 작성할 때는 맥락 정리에 시간이 걸림. 템플릿화하면 이후 재사용 가능.
결과물 검토·보정 시간 — AI 결과물은 항상 디자이너가 검토. 브랜드 기준 위반 여부, 디자인 시스템 준수 여부 확인 필요.
FE 협의 시간 — AI로 만든 UI가 코드 구조를 복잡하게 만드는 경우 별도 협의 필요. 초기엔 FE와 함께 검토하는 시간을 포함.
히스토리 기록 시간 — 어떤 프롬프트를 썼는지, 어떤 결과물이 나왔는지 기록. 팀 자산화까지 이어지려면 기록 시간을 일정에 포함.

04

가장 중요한 원칙 3가지

원칙 1 — AI에게 결과만 요청하지 말고 기준을 제공해야 한다

프롬프트보다 context 문서가 먼저입니다. AI에게 기준 문서를 제공할 때 결과물의 일관성이 높아집니다.

원칙 2 — AI가 반복해서 실수하는 부분을 문서로 만든다

AI가 매번 틀리는 것을 매번 프롬프트로 설명하면 비효율적입니다. 반복되는 기준은 문서로 빼야 합니다.

시맨틱 토큰 레이어

색상값을 직접 쓰는 대신 목적 기반 토큰으로 추상화합니다. 테마가 바뀌어도 컴포넌트 코드는 그대로 유지됩니다.

하지 말 것

color: #090C32;
border: 1px solid #E0E2ED;
이렇게
background: var(--bg-card);
color: var(--text-primary);
border: 1px solid var(--border);

:root {

  --bg-code: #ECEDF5;

  --border: #E0E2ED;
  --border-light: #ECEEF5;

  /* 텍스트 */
  --text-primary: #090C32;
  --text-secondary: #464A65;
  --text-muted: #9398AC;

전환 방식

  • 기본값 — prefers-color-scheme: dark 시스템 설정 따름
  • 수동 전환 — 사용자가 토글 시 localStorage에 저장, 이후 방문 시 유지
  • FOUC 방지 — <head> 안 인라인 스크립트로 렌더 전 테마 즉시 적용
HTML <head> — FOUC 방지
<script>
(function(){
  var t = localStorage.getItem('theme')
       || (window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light');
  document.documentElement.setAttribute('data-theme', t);
})();
</script>

# 브랜드 토큰 정의

## 컬러
- Primary: var(--accent) — #664FFF
- 임의 컬러 추가 금지. 새 컬러 필요 시 디자이너 승인 후 토큰 추가

## 타이포그래피
- 본문: Pretendard 16px / line-height 1.75
- 헤딩: font-weight 700, letter-spacing -0.02em
- 코드: JetBrains Mono 13px

## 스페이싱
- 기준 단위: 4px (4, 8, 12, 16, 20, 24, 32, 40, 48, 64)
- 임의 px 값 금지

## 반경
- 카드: 10px (--radius)
- 인라인: 6px (--radius-sm)
- 알약형: 20px

AI 결과물 브랜드 체크리스트

  • 색상이 모두 토큰 변수로 지정되어 있는가
  • 폰트가 정의된 스택만 사용하는가
  • 간격이 스페이싱 단위 기준인가 (4의 배수)
  • 기존 컴포넌트를 재사용했는가, 새로 만들었다면 이유가 있는가
  • 다크모드에서도 동일하게 읽히는가
  • 이 화면이 우리 제품처럼 보이는가 — 마지막 감각적 판단
AI Workflow 문서는 AI에게 일을 시키기 위한 프롬프트 모음이 아니라, AI가 제품과 디자인 시스템을 이해하고 일관되게 작업하도록 만드는 운영 체계다.

프로덕트 디자이너가 집중해야 할 부분은 코드를 직접 잘 짜는 것이 아니라, AI가 흔들리지 않고 디자인 판단을 보조할 수 있도록 제품 맥락, UX 의도, 컴포넌트 사용 기준, 검토 기준을 구조화하는 것입니다.
