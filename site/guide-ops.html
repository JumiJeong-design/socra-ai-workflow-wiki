03A

도구 / 작업 선택 가이드

AI 도구는 기능 목록으로 외우기보다, 실제 업무 흐름 안에서 선택하는 것이 좋습니다. 먼저 지금 하려는 일이 탐색, 정리, 검증, 전달 중 어디에 가까운지 판단하고 그에 맞는 작업 방식을 고릅니다.

핵심 기준: Figma와 AI 도구는 디자인 판단을 대신하는 도구가 아니라, 이미 정한 제품 맥락·UX 원칙·디자인 시스템 기준을 빠르게 적용하고 검토하기 위한 운영 도구입니다.

업무 흐름별 선택 순서

1. 탐색 / 초안 — 화면 구조나 플로우를 빠르게 보고 싶다면 Figma 시안, FigJam 플로우, 코드 스파이크 중 가장 빠른 형식으로 시작
↓
2. 시스템 적용 — 확정에 가까운 화면은 기존 컴포넌트, token, typography, spacing 기준에 맞춰 정리
↓
3. 드리프트 점검 — 로컬 스타일, override, 누락 token, 컴포넌트 이탈을 체크리스트나 AI 리뷰로 확인
↓
4. 스펙화 — 컴포넌트 릴리즈나 핸드오프 전에는 구조, 상태, 색상, 접근성 기준을 문서화
↓
5. 전달 / 동기화 — 코드 컴포넌트 연결, 토큰 차이, QA 결과를 개발팀이 확인 가능한 형식으로 정리

디자이너가 자주 만나는 상황별 추천

상황 먼저 할 일 다음에 이어질 작업
신규 기능 화면을 빠르게 구체화해야 한다 제품 맥락과 UX 원칙 기준으로 초안 생성 product-context.md, ux-principles.md 기준으로 초안을 만들고, Figma에서 방향성을 좁힙니다.
기존 화면이 디자인 시스템과 어긋나 보인다 컴포넌트, token, local style 이탈 점검 문제 지점을 확인한 뒤 component/token 기준에 맞게 정리합니다.
컴포넌트 업데이트를 개발팀에 넘겨야 한다 상태, variant, 접근성, 사용 조건 문서화 필요에 따라 구조, 색상, 문구, 상태값, 접근성 기준을 세부 스펙으로 보강합니다.
정책, 상태, 예외 케이스가 복잡하다 플로우 다이어그램으로 먼저 정리 FigJam에서 플로우를 합의한 뒤 화면 설계와 QA 체크리스트로 연결합니다.
코드와 Figma 기준이 서로 다를 수 있다 token/컴포넌트 drift 확인 토큰 drift를 확인하고, 변경 영향도가 크면 review / release 문서에 남깁니다.
디자인 리뷰나 공유 자료가 필요하다 결정 배경과 비교안을 공유 자료로 정리 결정 배경, 비교안, QA 결과를 슬라이드로 정리해 팀 공유 자료로 만듭니다.

ai-workflow.md
## Figma / AI 작업 선택 기준
- 탐색 단계에서는 Figma 시안, FigJam 플로우, 코드 스파이크 중 가장 빠른 방식으로 방향을 본다.
- 확정 단계에서는 component, token, typography, spacing 기준을 확인한다.
- 컴포넌트 릴리즈 전에는 구조, 상태, 색상, 접근성 스펙을 남긴다.
- 토큰 변경은 Figma와 코드의 drift를 확인한 뒤 반영한다.
- 복잡한 정책과 상태 변화는 화면을 만들기 전에 다이어그램으로 먼저 시각화한다.
- 최종 산출물은 review-checklist.md와 handoff.md 기준으로 사람이 승인한다.

방향 이탈(Drift) 감지 신호

AI를 쓰면 더 빨리 만들 수 있지만, 더 빨리 엉뚱한 방향으로 갈 수도 있습니다. 리뷰 막판에 품질을 고치는 것과 별개로, 작업 중간에 방향이 틀어지고 있는 신호를 먼저 잡아야 합니다.

신호 의미 대응 액션
피드백이 같은 질문을 맴돈다 디테일 문제가 아니라 방향 옵션이 부족한 상태 세부 조정을 멈추고 Figma 시안 또는 코드 스파이크 중 가장 빠른 방식으로 2~3개 대안을 새로 만든다
뭔가 이상하지만 설명하지 못한다 시각 품질, 정보 위계, 제품 톤이 섞여 감각적으로 어긋난 상태 Figma AI 에이전트나 감사 체크리스트로 캔버스 전체를 점검한다
빌드가 초기 제안에서 멀어진다 구현 과정에서 원래 UX 의도나 비교 기준이 흐려진 상태 원본 시안과 구현 화면을 나란히 두고 차이를 명시한다
팀이 방향을 고르지 못한다 선택 기준보다 취향 논의가 앞서는 상태 각 안이 얻는 것과 포기하는 것을 적고, 실제 작동 가능한 대안으로 비교한다

방향 확정 전 체크

1. 옵션을 나란히 비교 — 하나씩 말로 설명하지 말고 같은 조건에서 비교한다
↓
2. 트레이드오프를 명시 — 무엇이 좋아졌고 무엇을 포기하는지 적는다
↓
3. Figma 프로토타입 또는 코드로 현실화 — 실제 인터랙션과 제약 조건에서 무너지는지 확인한다
↓
4. 외부인도 이해 가능한지 확인 — 설명 없이 화면만 봐도 의도가 읽히는지 본다

충실도별 프로토타입 선택 기준

AI로 모션 시안 만들 때의 프롬프트 흐름

모션은 말로만 설명하면 오해가 큽니다. 레퍼런스 영상, 적용할 화면 캡처, 시간 조건을 함께 주고 한 단계씩 좁혀가는 방식이 더 안정적입니다. 목표는 AI가 최종 모션을 대신 확정하는 것이 아니라, 디자이너가 빠르게 비교하고 개발자에게 설명 가능한 산출물로 정리하는 것입니다.

쓸 만한 상황: Figma 프로토타입만으로 원하는 움직임을 만들기 어렵거나, 모션 디자이너 없이 FE와 “이런 느낌”을 공유해야 할 때. 특히 에러 강조, CTA 피드백, 차트 등장, 입력 누락 안내처럼 짧은 UI 모션에 적합합니다.

1. 레퍼런스 확보 — 비슷한 모션을 화면 녹화하거나, 모션 이름을 모르면 AI에게 먼저 이름을 묻는다
↓
2. 적용 화면 캡처 — 실제 적용될 UI를 함께 제공해 위치와 대상 요소를 명확히 한다
↓
3. 순서를 넘버링 — A에서 B로 이동, 100ms 후 강조, 1회만 재생처럼 단계와 시간을 수치로 적는다
↓
4. 조정 가능한 미리보기 요청 — 재생 버튼과 duration/delay 슬라이더를 넣어 디자이너가 감각을 조정하게 한다
↓
5. CSS 코드와 녹화본 전달 — 최종 모션은 CSS 기준 코드, 스크린 녹화, 의도 설명을 함께 개발팀에 넘긴다

모션 프롬프트에 꼭 넣을 것

항목 넣어야 하는 이유 예시
대상 요소 AI가 화면 전체를 임의로 바꾸는 것을 막기 위해 배송지 정보 타이틀과 라벨만 움직임
상태 변화 모션이 어떤 UX 피드백인지 명확히 하기 위해 미입력 상태에서 CTA 클릭 → 해당 영역 강조
시간값 수정 요청을 감각어가 아닌 수치로 주기 위해 delay 100ms, duration 300ms, 1회 재생
움직임 범위 과한 움직임이나 레이아웃 깨짐을 막기 위해 위로 4px 이동 후 제자리로 돌아옴
조정 UI 디자이너가 직접 속도를 비교하며 결정하기 위해 재생 버튼, duration/delay 슬라이더 포함
최종 전달 형식 FE가 바로 검토할 수 있는 언어로 넘기기 위해 CSS 코드, keyframes, 녹화본, 의도 설명
motion prompt
첨부한 화면에서 미입력 상태의 배송지 영역을 강조하는 모션을 CSS 기반으로 만들어주세요.

순서는 아래와 같습니다.
1. CTA 클릭 후 미입력 영역으로 scroll to
2. 배송지 정보 타이틀과 라벨 UI가 위로 4px 이동
3. 살짝 튕기는 느낌으로 제자리로 돌아옴
4. 모션은 1회만 재생
5. 디자이너가 duration과 delay를 조정할 수 있도록 재생 버튼과 슬라이더를 함께 제공

최종 결과는 CSS 코드로 전달할 수 있게 정리해주세요.

FE 전달 패키지

디자이너가 준비할 것
  • 최종 모션 녹화본 또는 GIF
  • 어떤 상태에서 재생되는지 조건
  • duration, delay, easing 값
  • 움직이는 요소와 움직이면 안 되는 요소
개발자와 확인할 것
  • 실제 코드 스택에 맞는 구현 방식 (CSS, React, emotion 등)
  • 반응형에서 위치가 깨지지 않는지
  • 접근성 설정인 prefers-reduced-motion 처리
  • 재사용 가능한 motion token 또는 utility로 뺄지 여부
⚠️

주의: AI가 만든 미리보기는 종종 원래 UI를 자기 방식으로 다시 그립니다. 모션을 볼 때는 화면 디자인이 아니라 움직임의 타이밍, easing, 방향성만 판단하고, 실제 적용은 기존 컴포넌트와 코드 구조 안에서 다시 검수합니다.

2주 실험 실행 계획

AI Workflow 도입의 효과를 확인하기 위한 2주간 실험 계획입니다. 결과물이 아니라 과정이 보이는 자료를 남기는 것이 핵심입니다.

👤

AI 활용 프로젝트 메인 담당자: 프로덕트 디자인팀 담당자가 진행 방식 논의, 회의 주체, 공유, 업무 팔로업을 담당합니다. 파트원이 여럿인 경우 별도 지정합니다.

실험 12단계 프로세스

  1. 현재 디자인 업무에서 부담이 큰 지점 정리
    기획서 확인 → 화면 설계 → 시안 제작 → 핸드오프 → QA 과정을 나열하고 손이 많이 가는 구간 정리. 담당: 디자이너, PM/PO, FE
  2. AI로 해결 가능한 업무와 불가능한 업무 구분
    화면 초안 생성, 반복 UI 제작 vs 브랜드 판단, UX 의사결정, 정책 해석. 담당: 디자이너, PM/PO
  3. 실험할 화면 또는 기능 선정
    신규 기능 초기 화면, 반복 카드 UI, 빈 상태/에러 상태 등 AI 적용 효과를 확인하기 좋은 화면 1~2개 선정. B2C 첫인상 구간은 AI 결과물을 그대로 쓰지 않고 디자이너가 강하게 보정. 담당: 디자이너, PM/PO
  4. 브랜드/UX/디자인 시스템 기준 먼저 정의
    톤앤매너, 컴포넌트 사용 기준, 컬러, 타이포, spacing, 피해야 할 느낌을 간단히 정리. 담당: 디자이너
  5. 기획서를 바탕으로 AI에 1차 화면 구조 생성 요청
    화면 구조, 정보 우선순위, 주요 섹션, 필요한 상태값, 예외 케이스 초안 생성. 완성된 디자인보다 화면 기획 보조와 구조화에 초점. 담당: 디자이너
  6. AI로 화면 초안 또는 UI 변형안 생성
    Claude, Codex 등 사용 가능한 도구로 화면 초안 생성. 반복 UI, 상태 화면, 리스트/카드/폼 구조에 AI 적극 활용. 담당: 디자이너, FE
  7. 디자이너가 UX/브랜드/디자인 시스템 관점으로 1차 보정
    정보 구조, 시선 흐름, 문구, 간격, 브랜드 톤 검수 및 수정. 디자인 시스템 기준을 벗어난 결과물은 수정 또는 재생성. 담당: 디자이너
  8. 프론트 개발자와 구현 가능성 검토
    컴포넌트 구조, 상태 관리, 반응형, API 연동 확인. AI UI가 코드 구조를 복잡하게 만들지 않는지 검토. 담당: FE, 디자이너
  9. 핸드오프 항목 체크리스트화
    컴포넌트 종류, 상태값, 반응형, 에러/로딩/빈 상태, 문구, 접근성, API 필요 여부를 AI가 초안 작성 후 디자이너·FE가 보완. 담당: 디자이너, FE
  10. 실제 구현 화면 기준으로 디자인 QA 진행
    Figma 시안이 아닌 실제 프론트 화면에서 간격, 정렬, 반응형, 문구, 상태, 브랜드 톤 확인. AI로 QA 항목 누락 여부 1차 점검. 담당: 디자이너, PM/PO, FE
  11. 작업 시간과 품질 변화 기록
    화면 초안 제작 시간, 핸드오프 작성 시간, QA 수정 횟수, 브랜드 보정 시간, FE 커뮤니케이션 횟수 기록. 담당: 메인 담당자, 디자이너
  12. 2주 후 결과 공유 및 적용 기준 정리
    AI를 계속 활용할 업무, 제한적으로 활용할 업무, 사용하지 않는 게 나은 업무를 구분. 담당: 전체

  • 어떤 화면/기능에 AI를 적용했는지
  • 어떤 도구를 사용했는지

  • 우리 팀 프로세스에 적용하려면 필요한 룰

AI 도입 시 디자이너 일정 산정 가이드

AI를 활용하면 일부 업무는 빨라지지만, 처음 도입하는 경우 익숙해지는 시간과 반복 검토 비용이 추가됩니다. AI 활용을 전제로 일정을 산정할 때는 다음 기준을 참고하세요.

⚠️

AI 도입 초기에는 일정을 단축하기보다 기존과 동일하게 잡는 것이 현실적입니다. 프롬프팅 비용, 결과물 검토, 보정 작업이 예상보다 클 수 있습니다.

업무 유형별 일정 조정 기준

단축 가능한 업무
  • 반복 UI 제작 — 카드, 리스트, 폼 등 구조가 반복되는 화면: 기존 대비 20~40% 단축 가능
  • 핸드오프 항목 정리 — 컴포넌트·상태값 목록 초안 생성: 30~50% 단축
  • 빈 상태 / 에러 상태 화면 — 패턴이 명확한 상태 화면: 30% 단축
  • 문서 초안 작성 — 체크리스트, 정책 요약, QA 항목: 40~60% 단축
기존 일정 유지 필요 업무
  • 브랜드 첫인상 구간 — 온보딩, 랜딩, 신규 기능 진입 화면: AI 결과물 그대로 사용 불가, 디자이너 직접 검토 필요
  • 복잡한 UX 흐름 — 결제, 인증, 다단계 플로우: AI 의존 시 논리 오류 위험
  • 신규 컴포넌트 설계 — 기존 패턴이 없는 첫 설계: AI 참고는 가능하나 판단은 디자이너 직접

일정 산정 시 추가로 고려할 항목

프롬프트 작성 시간 — 처음 작성할 때는 맥락 정리에 시간이 걸림. 템플릿화하면 이후 재사용 가능.
결과물 검토·보정 시간 — AI 결과물은 항상 디자이너가 검토. 브랜드 기준 위반 여부, 디자인 시스템 준수 여부 확인 필요.
FE 협의 시간 — AI로 만든 UI가 코드 구조를 복잡하게 만드는 경우 별도 협의 필요. 초기엔 FE와 함께 검토하는 시간을 포함.
히스토리 기록 시간 — 어떤 프롬프트를 썼는지, 어떤 결과물이 나왔는지 기록. 팀 자산화까지 이어지려면 기록 시간을 일정에 포함.

일정 산정 예시 — 신규 기능 화면 1개

업무 기존 일정 AI 활용 일정 비고
화면 구조 기획 0.5일 0.5일 AI로 초안 생성, 디자이너 검토·수정
시안 제작 (반복 UI) 1일 0.5~0.7일 카드·리스트 등 반복 구조 AI 생성
브랜드/UX 보정 포함 +0.3일 AI 결과물 품질 편차 보정 시간 추가
핸드오프 준비 0.5일 0.3일 체크리스트 AI 초안 생성 후 확인
FE 협의 0.5일 0.5~0.7일 AI UI 구현 이슈 발생 시 추가 협의
합계 2.5일 2.1~2.7일 처음엔 기존과 유사, 숙련 후 단축
💡

일정이 단축되는 것은 AI 활용이 팀에 완전히 정착된 이후입니다. 처음 2~4주는 학습 곡선을 감안해 기존 일정을 유지하고, 반복 업무에서 실제로 단축되는 경험이 쌓인 다음 일정 기준을 조정하세요.

04

가장 중요한 원칙 3가지

원칙 1 — AI에게 결과만 요청하지 말고 기준을 제공해야 한다

프롬프트보다 context 문서가 먼저입니다. AI에게 기준 문서를 제공할 때 결과물의 일관성이 높아집니다.

원칙 2 — AI가 반복해서 실수하는 부분을 문서로 만든다

AI가 매번 틀리는 것을 매번 프롬프트로 설명하면 비효율적입니다. 반복되는 기준은 문서로 빼야 합니다.

AI가 primary button을 너무 많이 쓴다
  → ux-principles.md에 hierarchy 원칙 추가

AI가 임의 색상을 만든다
  → agent-rules.md에 hardcoded color 금지 추가

AI가 접근성을 빼먹는다
  → review-checklist.md에 accessibility 항목 추가
원칙 3 — 문서는 완성물이 아니라 운영하면서 진화하는 기준이다

처음부터 완벽한 문서를 만들 필요는 없습니다. 실제 작업 중 생기는 문제를 기준화하는 흐름이 중요합니다.
