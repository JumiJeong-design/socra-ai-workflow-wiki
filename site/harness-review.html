문서 역할맵과
plan 번호화 기준

git 문서를 볼 때 “이 문서가 현재 기준인지, 규칙인지, 실행 절차인지, 과거 기록인지” 빠르게 구분하기 위한 번호 체계입니다.

이것만 기억
00 / 10
처음 보는 문서와 현재 상태
`00`은 입구, `10`은 현재 진행 중인 상태입니다. 지금 뭘 봐야 하는지는 이 두 번호에서 시작합니다.
20 / 30 / 40
규칙, 절차, 검증
`20`은 반드시 지킬 규칙, `30`은 반복 워크플로, `40`은 테스트와 리뷰 기준입니다.
50 / 60 / 70 / 80 / 90
계약, 템플릿, 기록, 공유, 아카이브
`50`은 source of truth, `60`은 prompt/skill, `70`은 raw log, `80`은 외부 공유, `90`은 과거 자료입니다.
번호 체계
00-09
Entry / Index: README, AGENTS, 문서 역할맵처럼 처음 들어오는 문서
10-19
Current State: 현재 상태, active plan, 이번 주 기준
20-29
Rules / Guardrails: 반드시 지켜야 하는 canonical rule
30-39
Playbooks / Workflows: 특정 상황에서 실행하는 절차
40-49
QA / Eval / Harness Checks: 자동 검증, 회귀 케이스, 리뷰 기준
50-59
Source of Truth / Contracts: 토큰, 컴포넌트, 제품 판단 계약
60-69
Templates / Prompts / Skill Entrypoints: 사람이 기억할 진입 스킬과 요청 템플릿
70-79
Logs / Raw Records: 날짜별 원본 기록. 현재 판단 기준으로 바로 쓰지 않음
80-89
External / Shareable: 외부 공유용으로 정제한 문서
90-99
Archive / Deprecated: 과거 방향, 보류 메모, 참고 이력
repo별로 이렇게 읽기
jumi-worklog
raw 기록과 현재 스냅샷
`CONTEXT.md`는 10번대 current snapshot 역할입니다. `logs/YYYY/MM`은 70번대 raw log입니다. 여기서는 원본 맥락을 확인합니다.
Prism
제품/디자인 시스템 계약
`design-system/rules.md`는 20번대 규칙, `packages/prism/*contract*`는 50번대 SOT입니다. `docs/plans`는 번호 prefix로 현재/과거를 나눕니다.
workflow-wiki
정제된 운영 지식과 HTML 표면
guide/playbook은 30/40/80/90 번호로 읽고, 주미님이 직접 보는 표면은 `site/*.html` 웹 페이지로 둡니다.

이름을 유지하는 문서

도구가 자동으로 찾는 파일은 번호를 붙이지 않는다.

`README.md`, `AGENTS.md`, `CLAUDE.md`는 root entry라 파일명을 유지합니다. `skills/**`도 discovery 경로와 연결될 수 있어서 rename하지 않습니다. 대신 내부에서 역할맵이나 canonical rule로 링크합니다.

주미님이 git 문서를 볼 때 순서
1
먼저 이 페이지에서 번호 의미를 확인한다.
2
repo의 `README.md` 또는 `AGENTS.md`로 들어간다.
3
현재 상태는 `10`, 규칙은 `20`, 절차는 `30`, 검증은 `40`, 계약은 `50` 문서에서 찾는다.
4
`70`이나 `90` 문서는 원본 기록/과거 참고로만 보고, 현재 기준으로 바로 실행하지 않는다.
